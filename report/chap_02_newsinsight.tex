\chapter{Global Architecture}

\section{Introduction}
Chapter 1 introduced News Insight's objectives and the technology stack chosen to meet them. This chapter assembles those pieces into a single system view. It first walks through the high-level architecture, from the news sources on the left to the React dashboard on the right, then details three complementary UML views: the use case diagram (who does what), the class diagram (what the system stores), and three sequence diagrams that trace the concrete request flows behind the platform's core features. Later chapters each zoom into one box of the diagrams introduced here: Chapters 3 and 4 detail the sources/storage boundary, Chapter 5 the ML enrichment step, Chapter 6 the GenAI layer, Chapter 7 the orchestration and tracking box, Chapter 8 the API, Chapter 9 the deployment, and Chapter 10 the frontend.

\section{High-level architecture}
\label{sec:high-level-arch}
Figure~\ref{fig:arch-highlevel} shows the system end to end. Data flows left to right through five stages, with two supporting services (Prefect and MLflow) wrapped around the middle of the pipeline.

\textbf{Sources.} Two collection mechanisms feed the pipeline. \texttt{RSSScraper} polls the RSS feeds of four Tunisian outlets — \textit{La Presse}, \textit{Business News}, \textit{Kapitalis}, and \textit{Mosaïque FM} (\texttt{etl/extract.py}, list \texttt{RSS\_FEEDS}) — and is the active default for every source. \texttt{KapitalisScraper}, a \texttt{BeautifulSoup}-based HTML listing scraper built on the common \texttt{BaseScraper} class (\texttt{scrapers/base.py}), is kept available as a deeper-crawl fallback for Kapitalis but is not wired into the default extraction run. Both paths converge on the same \texttt{Article} record shape (url, source, title, content, language, categories, published\_at, image\_url) before anything is written to the database.

\textbf{Storage.} A single PostgreSQL instance, with the \texttt{pgvector} and \texttt{unaccent} extensions enabled, holds the whole corpus in one \texttt{articles} table (\texttt{db/schema.sql}). \texttt{pgvector} lets the same table carry a 384-dimension sentence embedding per row and be queried by vector distance; \texttt{unaccent} lets the same table be queried by accent-insensitive French full text. Centralizing storage this way — rather than splitting raw and enriched data into separate tables — keeps the enrichment step a set of in-place column updates instead of a multi-table join.

\textbf{Enrichment.} From storage, four enrichment operations read and write back to the \texttt{articles} table: multilingual sentence embeddings (\texttt{intfloat/multilingual-e5-small}), sentiment classification (\texttt{cardiffnlp/twitter-xlm-roberta-base-sentiment}), unsupervised topic clustering (BERTopic), and LLM-based tagging for theme and region. In the running system these are not four independent parallel jobs: embeddings, sentiment, and theme classification happen together, per article, inside \texttt{transform\_articles} (\texttt{etl/transform.py}), and region is resolved by a gazetteer lookup of Tunisian governorate names in the same step (an LLM-based resolver, \texttt{tag\_region\_llm}, exists in \texttt{etl/regions.py} but is disabled by default for cost and latency). BERTopic clustering is the exception: it needs a batch of already-embedded articles to fit a topic model, so it runs as a separate, on-demand stage — triggered by an operator rather than on every new article — that overwrites \texttt{topic\_id}/\texttt{topic\_label} with a data-driven clustering once at least ten articles are available. Chapter 5 details each of these models and their trade-offs.

\textbf{Orchestration and tracking.} Prefect wraps the scrape-transform-load sequence in a single flow, \texttt{news-pipeline} (\texttt{flows/news\_flow.py}), whose one task retries twice with a 60-second delay on failure. The flow can be triggered on demand from the operator dashboard (\texttt{POST /pipeline/trigger}) or run on a daily schedule (06:00, Africa/Tunis, \texttt{flows/schedule.py}). Every BERTopic run is separately logged to MLflow under the \texttt{news-clustering} experiment (\texttt{etl/cluster.py}): parameters (embedding model, minimum cluster size), metrics (number of topics found, outlier percentage), and the fitted model itself as an artifact. Chapter 7 covers both services in detail.

\textbf{API and frontend.} A FastAPI application (\texttt{api/main.py}) exposes the enriched corpus through five routers — pipeline, articles, search, genai, and img — described in Chapter 8. Two of them, \texttt{search} and \texttt{genai}, call out to an external LLM provider (Groq by default, reached through an OpenAI-compatible client, as introduced in Chapter 1) for AI-ranked search fallback, article summaries, and the chat assistant. A React single-page application, built with Vite and served through nginx in production, is the only client of this API; nginx proxies every \texttt{/api/} request straight to the FastAPI container (Chapter 9, Chapter 10).

\section{Actors and use cases}
Two actors interact with the platform (Figure~\ref{fig:usecase}). The \textbf{Visitor} is anyone browsing the public dashboard: they can browse the article list and read a single article (\texttt{Articles.jsx}, \texttt{Reader.jsx}, \texttt{GET /articles}), run a semantic search (\texttt{Search.jsx}, \texttt{GET /search}), ask a question of the ``ask the news'' assistant (\texttt{ChatWidget.jsx}, \texttt{POST /genai/chat}), and view the topic, sentiment, and region dashboard (\texttt{Dashboard.jsx}, \texttt{Carte.jsx}). The \textbf{Operator} is a team member operating the pipeline: in addition to every visitor use case, they can trigger a new collection run or a clustering run (\texttt{Pipeline.jsx}, \texttt{POST /pipeline/trigger} and \texttt{POST /pipeline/cluster}) and monitor past pipeline runs from the same page's status panel. The distinction is one of intent rather than of authentication — both roles use the same public frontend — but it separates the platform's two concerns cleanly: consuming the enriched news corpus, and operating the pipeline that produces it.

\section{Domain model}
Figure~\ref{fig:classes} gives the conceptual class view behind the single \texttt{articles} table. \textbf{Article} is the central entity: it carries source metadata (url, source, title, content, language, timestamps, image, categories) and the enrichment fields written by the pipeline described above — the 384-dimension embedding, the sentiment label, and \texttt{story\_id} (reserved for a future cross-source story-clustering feature, not yet populated by the pipeline). Three further concepts relate to an article but are not separate database tables: \textbf{Topic/Theme} (a topic id and a human-readable label, set either by the LLM theme classifier or overwritten by BERTopic clustering), \textbf{Region} (the Tunisian governorate an article is tagged with, exposed on the map dashboard), and \textbf{Summary} (the on-demand, cached LLM summary produced by \texttt{GET /genai/summary/\{id\}}). Each of the three is modelled as a zero-or-one relationship from Article because each maps directly onto a nullable column set on the \texttt{articles} table (\texttt{topic\_id}/\texttt{topic\_label}, \texttt{region}, \texttt{summary}) rather than a foreign-key relation — a deliberate flattening that keeps every read a single-table query, at the cost of not normalizing region or topic metadata. The full schema also defines two operational tables outside this conceptual model, an \texttt{entities} table for named-entity extraction and a \texttt{pipeline\_metrics} table for run monitoring, covered where they are used in Chapters 4 and 7.

\section{Key scenarios}
The three sequence diagrams below trace the request flows behind News Insight's three core features, each spanning the layers introduced in Section~\ref{sec:high-level-arch}: the background collection pipeline, and the two GenAI-backed, user-triggered flows built on top of it.

\subsection{Scrape, transform, and load pipeline}
Figure~\ref{fig:seq-pipeline} traces one run of the \texttt{news-pipeline} Prefect flow. The \texttt{pipeline\_task} calls \texttt{run\_pipeline} (\texttt{etl/pipeline.py}), which runs its three steps strictly in sequence: \texttt{run\_extraction} collects raw \texttt{Article} objects from the RSS and HTML scrapers; \texttt{transform\_articles} computes embeddings, sentiment, and theme/region tags for the whole batch; and \texttt{save\_articles} upserts the enriched rows into PostgreSQL with \texttt{INSERT ... ON CONFLICT (url) DO UPDATE}, so re-running the flow on already-seen URLs is safe and refreshes their enrichment rather than duplicating rows. The pipeline then records aggregate counters (articles saved, sentiment breakdown, embedding model used) to \texttt{pipeline\_metrics} before returning. If any step raises, Prefect's task-level retry (two attempts, 60-second delay) re-runs the whole task rather than resuming mid-pipeline.

\subsection{AI-assisted semantic search}
Figure~\ref{fig:seq-aisearch} traces a \texttt{GET /search} request (\texttt{api/routes/search.py}). The router first checks, cheaply and offline, whether the embedding model's weights are already cached (\texttt{embeddings\_available()}); this avoids ever blocking a search request on a multi-minute model download. If the model is available, the query is embedded and the router asks PostgreSQL to order every article by \texttt{embedding <=> query\_vector} (pgvector's cosine-distance operator), returning the closest matches with a similarity score. If not, the router falls back to a two-stage keyword-plus-LLM path: it collects candidates from a French full-text match (\texttt{tsvector}/\texttt{websearch\_to\_tsquery}, accent-folded) topped up with recent articles, then asks the LLM to re-rank and filter that candidate pool down to the genuinely relevant ones (\texttt{\_ai\_rank}), so a query like ``sport'' can still surface an article about a football club acronym that the keyword match alone would miss. Either path returns the same result shape to the frontend.

\subsection{Ask-the-news chat (retrieval-augmented generation)}
Figure~\ref{fig:seq-chat} traces a \texttt{POST /genai/chat} request (\texttt{api/routes/genai.py}). It follows the same cached-embedding check as search, retrieving the five closest articles by vector distance when the embedding model is available, or the five highest-ranked by French full-text relevance (falling back to recency) otherwise — always five, so the LLM has real, current articles to ground its answer even on a loose or no-match query. The router then builds a numbered context block from those five articles' titles and excerpts and sends it, together with the user's question, to the LLM with an instruction to answer strictly from the supplied context and cite source numbers. The response returned to the frontend pairs the generated answer with the list of source articles (id, title, url) it was built from, which \texttt{ChatWidget.jsx} renders as clickable citations — this retrieval-then-generate structure is what keeps the assistant's answers traceable back to actual News Insight articles rather than the model's unverified prior knowledge.

\section{Conclusion}
This chapter moved from the technology choices of Chapter 1 to a concrete system view: a five-stage pipeline from Tunisian news sources to a React dashboard, orchestrated by Prefect and tracked by MLflow, sitting on a single PostgreSQL/pgvector table that keeps every enrichment step a plain column update. The use case, class, and sequence diagrams gave three complementary views of the same system — who uses it, what it stores, and how a request actually flows through it — that the next chapters revisit one layer at a time, starting with the data sources and scraping stage in Chapter 3.
