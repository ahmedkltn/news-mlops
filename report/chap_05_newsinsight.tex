\chapter{Machine Learning and NLP Pipeline}

\section{Introduction}
\begin{sloppypar}
Chapter 4 closed by following the \texttt{embedding}, \texttt{sentiment}, \texttt{topic\_id}\allowbreak/\allowbreak\texttt{topic\_label}, and \texttt{region} columns of the \texttt{articles} table back to ``the machine learning pipeline that actually computes them.'' This chapter is that pipeline. It reads six files end to end -- \texttt{etl/\allowbreak transform.py}, \texttt{etl/\allowbreak cluster.py}, \texttt{etl/\allowbreak themes.py}, \texttt{etl/\allowbreak labels.py}, \texttt{etl/\allowbreak regions.py}, and \texttt{etl/\allowbreak metrics.py} -- and, together with the orchestration entry point \texttt{etl/\allowbreak pipeline.py}, explains exactly which model computes which column, with which parameters, and in which order. Three model families do the heavy lifting: a multilingual sentence-transformer for semantic embeddings, a fine-tuned XLM-RoBERTa classifier for sentiment, and BERTopic for unsupervised topic discovery, each wrapped by lighter, code-level heuristics (an LLM theme classifier, an LLM topic-labeller, and a gazetteer-first region tagger) that turn raw model output into the columns Chapter 4 described. Figure~\ref{fig:ml-pipeline} is the map referred to throughout the chapter: the per-article transform stage on top, the periodic clustering stage below it, and the PostgreSQL \texttt{articles} table both stages write to.
\end{sloppypar}

\section{Pipeline overview}
\label{sec:pipeline-overview}
\texttt{run\_pipeline} (\texttt{etl/\allowbreak pipeline.py}) is the single entry point Chapter 2's sequence diagram (Figure~\ref{fig:seq-pipeline}) already traced: four steps, run strictly in sequence, with no step started before the previous one returns. \textbf{Extract} (\texttt{run\_extraction}, Chapter 3) collects raw \texttt{Article} objects from the RSS and HTML scrapers; if it returns nothing, the function logs a warning and returns immediately with a zeroed result dictionary, so nothing downstream ever runs on an empty batch. \textbf{Transform} (\texttt{transform\_articles}, Section~\ref{sec:embeddings}) is where embeddings, sentiment, theme, and region are all computed for the whole batch. \textbf{Load} (\texttt{save\_articles}, Chapters 3--4) upserts the enriched rows with \texttt{ON CONFLICT (url) DO UPDATE}. \textbf{Metrics} (\texttt{record\_pipeline\_metrics}, Section~\ref{sec:metrics}) writes one summary row and is wrapped in its own \texttt{try/except} so that a metrics-write failure is logged and swallowed rather than failing an otherwise successful run. The module-level constant \texttt{EMBED\_MODEL = "intfloat/\allowbreak multilingual-e5-small"} is defined here purely to be recorded on that metrics row; the model itself is loaded from a separate, identically-valued constant, \texttt{EMBED\_MODEL\_NAME}, inside \texttt{etl/\allowbreak transform.py} -- the two are kept in sync by hand rather than imported from one shared definition.

\begin{figure}[H]
\centering
\includegraphics[width=0.95\columnwidth]{img/ml-pipeline.png}
\caption{The ML/NLP pipeline: per-article transform (embeddings, sentiment, theme, region) feeding \texttt{articles}, and the separate, periodic BERTopic clustering stage that reads embeddings back out and overwrites \texttt{topic\_id}/\texttt{topic\_label}.}
\label{fig:ml-pipeline}
\end{figure}

\texttt{run\_pipeline} itself is reachable three ways: directly (\texttt{python -m etl.pipeline}), wrapped in a Prefect task with two retries and a 60-second delay (\texttt{flows/news\_flow.py}, Chapter 7), or triggered on demand as a FastAPI background task (\texttt{POST /pipeline/trigger}, \texttt{api/routes/pipeline.py}). Notably, BERTopic clustering (Section~\ref{sec:bertopic}) is \emph{not} one of its four steps: it needs a batch of already-embedded articles and enough of them to fit a topic model, so it runs as a separate stage, invoked by an operator through \texttt{scripts/\allowbreak cluster.py} (\texttt{make cluster}, \texttt{MIN\_CLUSTER=3} by default) or \texttt{POST /pipeline/cluster}. A third, narrower entry point exists purely for local development: \texttt{scripts/\allowbreak transform.py} (\texttt{make transform}, \texttt{BATCH\_SIZE=16} by default) re-embeds and re-scores only the articles still missing an embedding (\texttt{fetch\_\allowbreak untransformed\_articles}, \texttt{etl/\allowbreak load.py}), writing embedding and sentiment back one row at a time through \texttt{update\_\allowbreak article\_ml\_fields} -- it deliberately skips theme and region entirely, unlike \texttt{transform\_articles}, which computes all four in one batch.

\section{Text embeddings}
\label{sec:embeddings}
\texttt{transform\_articles} opens by building one text per article -- \texttt{"passage: "} followed by the title and content, matching the e5 prefixing convention already introduced from the storage side in Chapter 4 -- and encoding the whole batch in a single call to \texttt{model.encode}, rather than one call per article (Listing~5.1). The embedding model, \texttt{intfloat/\allowbreak multilingual-e5-small} (\texttt{EMBED\_MODEL\_NAME}), is loaded once through a lazy singleton, \texttt{\_get\_embedding\_model}: a module-level \texttt{\_embedding\_model} variable is populated on first use and reused afterwards, so a script that only needs, say, region tagging never pays the cost -- a few hundred megabytes of weights -- of loading it, and the FastAPI process does not load it at startup either. \texttt{embeddings\_available()}, already described in Chapter 4 from the search side, is the same offline cache check reused here conceptually: the model is downloaded from the Hugging Face Hub on first real use if not already cached locally.

\begin{verbatim}
texts = [
    f"passage: {(a.title or '')} {(a.content or '')}".strip()
    for a in articles
]
embeddings = model.encode(
    texts,
    normalize_embeddings=True,
    batch_size=16,
    show_progress_bar=True,
)
\end{verbatim}
\begin{center}
\textit{Listing~5.1: batched e5 passage encoding (\texttt{transform\_articles}, \texttt{etl/\allowbreak transform.py}) -- one \texttt{model.encode} call for the whole run.}
\end{center}

\texttt{normalize\_embeddings=True} makes every stored vector unit-length, the property Chapter 4 relied on to make pgvector's \texttt{<=>} cosine-distance operator directly meaningful. \texttt{batch\_size=16} here is a value hard-coded inside \texttt{transform\_articles} itself, distinct from the Makefile's configurable \texttt{BATCH\_SIZE} variable, which instead controls the batch size passed to \texttt{scripts/\allowbreak transform.py}'s own, separate \texttt{model.encode} call (Section~\ref{sec:pipeline-overview}). Each resulting vector is 384-dimensional, fixed by the model architecture and matching the \texttt{vector(384)} column declared in \texttt{db/schema.sql}. The asymmetric prefix is what makes this a \emph{passage} embedding rather than a \emph{query} embedding: \texttt{get\_embedding}, the function \texttt{api/routes/search.py} calls on an incoming search string, prefixes with \texttt{"query: "} instead, so a stored article vector and a live search-query vector are comparable in the same trained embedding space even though they went through different code paths.

\section{Sentiment analysis}
\label{sec:sentiment}
\begin{sloppypar}
Sentiment is computed by \texttt{cardiffnlp/\allowbreak twitter-xlm-roberta-\allowbreak base-sentiment}, a fine-tuned XLM-RoBERTa model, loaded lazily the same way as the embedding model through \texttt{\_get\_\allowbreak sentiment\_pipeline}: a Hugging Face \texttt{transformers.pipeline("text-classification", ...)}, with \texttt{truncation=True} and \texttt{max\_length=512} tokens, and \texttt{device=0 if torch.cuda.is\_available() else -1} so the same code runs on GPU where available and falls back to CPU otherwise. \texttt{transform\_articles} calls the batched helper, \texttt{get\_sentiments}, on the plain \texttt{"\{title\} \{content\}"} text -- sentiment does not use the e5 prefix, which is specific to the embedding step. Each text is first truncated to its first 1000 characters (a cheap pre-truncation ahead of the model's own 512-token limit), and the whole list is scored in one \texttt{pipe(..., batch\_size=16)} call.
\end{sloppypar}

The model's predicted label is lower-cased and looked up in \texttt{SENTIMENT\_MAP}, a dictionary mapping \texttt{"positive"}, \texttt{"neutral"}, and \texttt{"negative"} to themselves, with \texttt{.get(label, "neutral")} so that any label the map does not recognize -- not just I/O or model errors -- silently resolves to \texttt{"neutral"} instead of propagating. This is deliberate defensive coding rather than a translation between two different label vocabularies: \texttt{get\_sentiments} additionally wraps the whole batched call in a \texttt{try/except} that defaults \emph{every} text in the batch to \texttt{"neutral"} on any failure, so a broken sentiment model degrades the run's sentiment quality rather than aborting it -- the same never-break-the-pipeline philosophy \texttt{run\_pipeline} applies to its own metrics write (Section~\ref{sec:pipeline-overview}). A singular counterpart, \texttt{get\_sentiment}, exists in the same module with identical truncation and mapping logic but is not called from anywhere else in the codebase as of this pipeline.

\section{Topic modeling with BERTopic}
\label{sec:bertopic}
BERTopic clustering (\texttt{run\_clustering}, \texttt{etl/\allowbreak cluster.py}) is the data-driven counterpart to the per-article theme classifier of Section~\ref{sec:enrichment}: instead of asking an LLM to place one article into one of ten fixed categories, it looks at the embeddings of \emph{every} article with one already assigned and discovers topics as clusters in that vector space. \texttt{load\_articles\_\allowbreak for\_clustering} pulls \texttt{id}, \texttt{title}, \texttt{content}, and \texttt{embedding} for every row where \texttt{embedding IS NOT NULL}; if fewer than ten articles qualify, the function logs a warning and returns \texttt{\{"status": "skipped", "reason": "not enough articles"\}} without fitting anything (Listing~5.2) -- BERTopic's own clustering step needs a minimum amount of data to find stable clusters at all.

\begin{verbatim}
if len(ids) < 10:
    logger.warning(
        f"Not enough articles: {len(ids)}, "
        "need at least 10"
    )
    return {"status": "skipped",
            "reason": "not enough articles"}
\end{verbatim}
\begin{center}
\textit{Listing~5.2: the minimum-corpus guard (\texttt{run\_clustering}, \texttt{etl/\allowbreak cluster.py}) -- clustering never runs on fewer than ten embedded articles.}
\end{center}

\begin{sloppypar}
Article text is vectorized for keyword extraction by a \texttt{CountVectorizer} with \texttt{min\_df=1} (a term needs to appear in only one document to be kept -- appropriate at the project's corpus size), French stop words from \texttt{spacy.lang.fr.stop\_words}, and \texttt{ngram\_range=(1, 2)} so single words and two-word phrases are both candidate topic keywords. \texttt{BERTopic} is then constructed with \texttt{vectorizer\_model=vectorizer}, \texttt{min\_topic\_size=\allowbreak min\_cluster\_size} (a \texttt{run\_clustering} parameter defaulting to 3, and exposed as the Makefile's \texttt{MIN\_CLUSTER} variable through \texttt{scripts/\allowbreak cluster.py}'s \texttt{--min-cluster-size} flag), \texttt{language="french"}, and \texttt{calculate\_probabilities=False}. No custom \texttt{umap\_model} or \texttt{hdbscan\_model} is passed in, so dimensionality reduction and density-based clustering both run with BERTopic's own internal defaults, applied on top of the e5 embeddings already computed by \texttt{transform\_articles} and passed straight in through \texttt{fit\_transform(texts, embeddings=\allowbreak embeddings\_array)} -- BERTopic never recomputes its own embeddings here, it clusters the ones already stored in \texttt{articles.embedding}.
\end{sloppypar}

\texttt{get\_topic\_info()} then returns one row per discovered cluster, each with an integer topic id and a raw, auto-generated \texttt{Name} (BERTopic's own keyword-concatenated string), excluding topic \texttt{-1} -- the reserved id for articles that did not fit any dense cluster (HDBSCAN's outlier bucket). These raw names, together with each topic's top keywords from \texttt{topic\_model.get\_topic(tid)}, are handed to \texttt{label\_topics} (Section~\ref{sec:enrichment}) to be turned into short, human-readable labels before anything is written back; the whole \texttt{label\_topics} call is wrapped in a \texttt{try/except} that falls back to the raw BERTopic names if label generation fails outright. The fitted model is persisted to \texttt{models/bertopic\_model} (\texttt{topic\_model.save(...)}), and \texttt{update\_topics} overwrites \texttt{topic\_id}/\texttt{topic\_label} for every clustered article -- including the outliers, which get \texttt{topic\_id=-1} and whatever label \texttt{topic\_labels.get(-1, "unknown")} resolves to, since \texttt{-1} was excluded from the labelled dictionary. This is a genuine overwrite of the theme values \texttt{transform\_articles} assigned earlier: an article's \texttt{topic\_id}/\texttt{topic\_label} pair reflects whichever ran most recently, LLM theming at ingestion time or BERTopic clustering afterwards. Finally, \texttt{outlier\_pct} is computed as the fraction of articles assigned topic \texttt{-1}, and \texttt{log\_clustering\_run} records \texttt{embed\_model} and \texttt{min\_cluster\_size} as MLflow parameters and \texttt{n\_topics}/\texttt{outlier\_pct} as MLflow metrics under the \texttt{news-clustering} experiment, with the saved model directory logged as an artifact -- the operational side of this run is covered in Chapter 7.

\section{Enrichment: themes, labels and regions}
\label{sec:enrichment}
Three lighter mechanisms fill in the remaining enrichment columns, each combining a small deterministic or LLM step with a hard-coded fallback.

\textbf{Themes.} \texttt{classify\_theme} (\texttt{etl/\allowbreak themes.py}) is what \texttt{transform\_articles} calls for every article, immediately, well before BERTopic clustering ever runs -- it is the theme mechanism referred to as ``the LLM theme classifier'' in Chapter 2. It sends the article's title and the first 400 characters of its content to a fast LLM with a French prompt naming exactly ten fixed categories (\texttt{THEMES}: Politique, Économie, Sport, Société, International, Culture, Sécurité, Santé, Environnement, Technologie) and asks for one category name back, capped at \texttt{max\_tokens=8}. The reply is matched against the fixed list by \texttt{\_match}, which normalizes accents and case and accepts either an exact match or a substring containment; any parsing failure, LLM error, or unmatched reply defaults to \texttt{"Société"}. \texttt{theme\_id} then converts the chosen name to the integer stored in \texttt{topic\_id} by taking its index in \texttt{THEMES} -- and, distinct from BERTopic's genuine \texttt{-1} outlier sentinel, falls back to \texttt{len(THEMES) - 1} (the index of \texttt{"Technologie"}, the last entry) for any name it does not recognize, rather than to a sentinel meaning ``no theme.'' The LLM call itself is imported locally inside \texttt{classify\_theme}, the same lazy-loading pattern used for the embedding and sentiment singletons in Section~\ref{sec:embeddings}. Provider and model-selection details behind this and the next LLM call are covered in Chapter 6.

\begin{sloppypar}
\textbf{Labels.} \texttt{label\_topics} (\texttt{etl/\allowbreak labels.py}) is what turns a BERTopic cluster's raw, keyword-concatenated name into something readable: for each topic id it joins up to eight of the cluster's top keywords and asks a fast LLM for a two-to-four-word French title with no closing punctuation, capped at \texttt{max\_tokens=24}. Each per-topic call is individually guarded -- a failure for one topic falls back to the literal string \texttt{f"Topic \{tid\}"} rather than aborting the whole labelling pass, which is itself wrapped again by \texttt{run\_clustering}'s outer fallback to the raw BERTopic names (Section~\ref{sec:bertopic}): two independent layers of degradation between a fully LLM-labelled cluster set and the model's own auto-generated names.
\end{sloppypar}

\textbf{Regions.} \texttt{etl/\allowbreak regions.py} holds a gazetteer of Tunisia's 24 governorates (\texttt{GOVERNORATES}): for each, a slug, French and Arabic names, an ISO 3166-2 \texttt{geojson\_id} for the map dashboard, and a list of aliases covering name variants and major cities or delegations. \texttt{normalize} strips Latin accents (Unicode NFD decomposition, dropping combining marks) and lower-cases text while leaving Arabic letters intact, and \texttt{ALIAS\_INDEX} flattens every alias across all 24 governorates, sorted longest-first so a multi-word place name is matched before a shorter substring of it. \texttt{tag\_region\_gazetteer} scores each governorate by counting word-boundary alias matches, weighting a title hit three times a body hit (\texttt{\_score\_regions}), and returns the strict single winner -- \texttt{None} if no alias matched at all, or if two or more governorates tie for the highest score, rather than guessing. A second tier, \texttt{tag\_region\_llm}, asks an LLM to pick one of the 24 slugs or \texttt{"national"} when the gazetteer alone is ambiguous, validating the reply against the known slug set before trusting it. \texttt{tag\_region(title, content, use\_llm=True)} chains the two: gazetteer first, LLM only if \texttt{use\_llm} is true and the gazetteer found nothing. The function's own default is \texttt{use\_llm=True}, but \texttt{transform\_articles} explicitly calls it with \texttt{use\_llm=False} -- so in the pipeline that actually runs, region tagging is gazetteer-only; an article the gazetteer cannot place unambiguously is simply left with \texttt{region = NULL}, shown on the frontend map as the ``National / non-localisé'' bucket (\texttt{region\_label(None)}), and \texttt{tag\_region\_llm} is fully implemented and tested but never exercised by this call site.

\section{Pipeline metrics}
\label{sec:metrics}
\texttt{record\_pipeline\_metrics} (\texttt{etl/\allowbreak metrics.py}) does one thing: insert one row into \texttt{pipeline\_metrics} (Chapter 4) with whatever dictionary it is given. Its only caller is \texttt{run\_pipeline}, at the very end of every full run, and the values it passes reveal where the two stages of this chapter do and do not talk to each other. \texttt{source} is always the fixed string \texttt{"all"} -- the column exists for a per-source breakdown the current code never produces, one row is written per full pipeline run regardless of how many distinct sources it scraped. \texttt{articles\_new} is the row count \texttt{save\_articles} returned. \texttt{sentiment\_pos}/\texttt{sentiment\_neu}/\texttt{sentiment\_neg} are counted directly off the in-memory \texttt{transformed} list by comparing each article's \texttt{sentiment} field. \texttt{embed\_model} is the \texttt{EMBED\_MODEL} constant introduced in Section~\ref{sec:pipeline-overview}. \texttt{n\_topics} and \texttt{outlier\_pct}, by contrast, are always \texttt{None} on this path -- \texttt{run\_pipeline} never runs BERTopic clustering (Section~\ref{sec:bertopic}), so it has no topic count or outlier rate to report. \texttt{run\_clustering} does compute both, but sends them to MLflow's \texttt{news-clustering} experiment via \texttt{log\_clustering\_run} instead: as of this pipeline, no code path ever inserts a \texttt{pipeline\_metrics} row with \texttt{n\_topics} or \texttt{outlier\_pct} populated, and the two columns Chapter 4 documented in the schema remain schema-only for clustering, tracked instead through MLflow -- the operational use of both destinations is picked up in Chapter 7.

\section{Conclusion}
\begin{sloppypar}
This chapter followed every enrichment column Chapter 4 introduced back to the model call or heuristic that fills it in: \texttt{intfloat/\allowbreak multilingual-e5-small} for 384-dimension, prefix-asymmetric, batch-encoded embeddings; \texttt{cardiffnlp/\allowbreak twitter-xlm-roberta-\allowbreak base-sentiment} for batched, fail-safe-to-neutral sentiment; a ten-category LLM classifier for the theme every article gets at ingestion time, overwritten on demand by BERTopic (French-stop-word vectorized, \texttt{min\_topic\_size=3} by default, guarded against fewer than ten articles, its raw cluster names relabelled by a second LLM pass, its \texttt{-1} bucket a genuine ``no cluster'' outlier distinct from the theme classifier's forced ten-way choice); and a gazetteer-first, LLM-fallback-disabled-by-default tagger over Tunisia's 24 governorates for region. A single \texttt{pipeline\_metrics} row closes out every full run, though its topic-related columns stay empty on that path, populated conceptually only by the separate MLflow tracking of clustering runs. The next chapter turns to the LLM layer these theme, label, and region calls all quietly depend on: the provider-agnostic client behind \texttt{etl/llm.py}, and the GenAI features -- summaries and the ``ask the news'' chat assistant -- built on top of it.
\end{sloppypar}
