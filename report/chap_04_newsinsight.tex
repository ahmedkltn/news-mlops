\chapter{Storage Layer and Database Design}

\section{Introduction}
Chapter 2's class diagram (Figure~\ref{fig:classes}) already sketched the conceptual model behind News Insight's storage: a single \textbf{Article} entity carrying source metadata and pipeline-computed enrichment, with \textbf{Topic/Theme}, \textbf{Region}, and \textbf{Summary} flattened onto it as nullable columns rather than separate tables, plus two operational tables, \texttt{entities} and \texttt{pipeline\_metrics}, sitting outside that conceptual model. This chapter opens the database itself: why PostgreSQL with the pgvector extension was chosen as the single engine for both relational metadata and dense vector embeddings, the full column-and-index layout of \texttt{articles} and its two auxiliary tables, how vector similarity search is actually executed against the \texttt{embedding} column, and how the schema grew from its original shape to today's through two small, additive migrations. Every table, column, and index described below is read directly from \texttt{db/schema.sql} and the two files under \texttt{db/migrations/}.

\section{Database technology choice}
\label{sec:db-tech}
News Insight needs to store two different kinds of data per article: structured metadata (source, title, timestamps, sentiment label, topic, region) that benefits from ordinary relational filtering and sorting, and a 384-dimension dense vector per article (Section~\ref{sec:pgvector}) needed for semantic similarity search. A common architecture for the second half is a dedicated vector database (Pinecone, Weaviate, Qdrant, Milvus) running alongside the relational store, with the application responsible for keeping the two in sync: every article would be written twice, once to each system, and a query that needs both a similarity rank and a relational filter (for instance, ``the most similar articles that are also recent'') would have to fan out to two systems and merge the results in application code.

News Insight instead uses PostgreSQL with the \texttt{pgvector} extension, already listed as the storage row of Chapter 1's technology table (Table~\ref{tab:tech-stack}). \texttt{pgvector} adds a \texttt{vector} column type and a family of distance operators directly inside Postgres, so the embedding is simply one more column on the \texttt{articles} table rather than a record in a second system. The extension is declared at the top of \texttt{db/schema.sql}, alongside \texttt{unaccent} (used by the accent-insensitive French full-text fallback in \texttt{api/routes/search.py}):

\begin{verbatim}
CREATE EXTENSION IF NOT EXISTS vector;
CREATE EXTENSION IF NOT EXISTS unaccent;
\end{verbatim}

This single-engine choice pays off directly in the codebase: \texttt{etl/load.py}'s \texttt{get\_connection()} is the one connection helper used both by the pipeline's writer (\texttt{save\_articles}) and, imported unchanged, by the API's reader (\texttt{api/routes/search.py}, \texttt{api/routes/genai.py}): no separate client, no second set of credentials, and a similarity query can sit in the same \texttt{SELECT} as an ordinary \texttt{WHERE embedding IS NOT NULL} clause (Section~\ref{sec:pgvector}). The corpus itself is a further argument for this choice: a few thousand articles collected periodically from four Tunisian sources (Chapter 3) is well within reach of pgvector's exact nearest-neighbor scan, without needing a purpose-built approximate nearest-neighbor (ANN) engine sized for corpora orders of magnitude larger. Finally, Postgres is already the one stateful service in the project's Docker Compose stack (Chapter 9); enabling pgvector costs nothing beyond an extension flag in the same container, with no additional service to deploy, monitor, or back up.

\section{Schema design}
\label{sec:schema}
Figure~\ref{fig:db-schema} shows the full storage layer as an entity-relationship diagram: the \texttt{articles} table at its center, the \texttt{entities} table related to it through a foreign key, and the standalone \texttt{pipeline\_metrics} table.

\begin{figure}[H]
\centering
\includegraphics[width=0.85\columnwidth]{img/db-schema.png}
\caption{Storage layer schema: \texttt{articles} (central table, with its pgvector embedding column and indexes), \texttt{entities} (one row per extracted named entity, foreign key to \texttt{articles}), and \texttt{pipeline\_metrics} (one row per pipeline run).}
\label{fig:db-schema}
\end{figure}

\texttt{articles} is the table every other part of the system reads from and writes to. Its core columns, read off \texttt{db/schema.sql}, are: \texttt{id} (\texttt{SERIAL PRIMARY KEY}); \texttt{url} (\texttt{TEXT UNIQUE NOT NULL}, the durable deduplication guarantee already described in Chapter 3); \texttt{source} and \texttt{language}; \texttt{title} and \texttt{content}; \texttt{scraped\_at} (\texttt{TIMESTAMP DEFAULT NOW()}) and \texttt{published\_at} (\texttt{TIMESTAMP}, parsed from the source); \texttt{image\_url} and \texttt{categories} (\texttt{TEXT[]}, a native Postgres array rather than a separate join table, since a source's category list is small and never queried by individual category); \texttt{summary} (the on-demand, cached LLM summary from Chapter 2); \texttt{story\_id} (\texttt{INTEGER}, reserved for a future cross-source story-clustering feature, not yet populated by any pipeline step); \texttt{region} (\texttt{TEXT}, added by migration 002, Section~\ref{sec:migrations}); and the three ML-enrichment columns written by \texttt{transform\_articles} (Chapter 5): \texttt{topic\_id}, \texttt{topic\_label}, \texttt{sentiment}, and \texttt{embedding vector(384)} (Section~\ref{sec:pgvector}). Listing~4.1 shows the table declaration itself.

\begin{verbatim}
CREATE TABLE IF NOT EXISTS articles (
    id           SERIAL PRIMARY KEY,
    url          TEXT UNIQUE NOT NULL,
    source       TEXT NOT NULL,
    title        TEXT,
    content      TEXT,
    language     TEXT,
    scraped_at   TIMESTAMP DEFAULT NOW(),
    topic_id     INTEGER,
    topic_label  TEXT,
    sentiment    TEXT,
    embedding    vector(384),
    published_at TIMESTAMP,
    image_url    TEXT,
    categories   TEXT[],
    summary      TEXT,
    story_id     INTEGER,
    region       TEXT
);
\end{verbatim}
\begin{center}
\textit{Listing~4.1: the \texttt{articles} table (\texttt{db/schema.sql}): one row per article, enrichment columns nullable until the pipeline fills them in.}
\end{center}

Six B-tree indexes back the queries the dashboard and API actually run: \texttt{idx\_\allowbreak articles\_\allowbreak topic} on \texttt{topic\_id} (topic dashboard), \texttt{idx\_\allowbreak articles\_\allowbreak source} on \texttt{source} (per-source browsing), \texttt{idx\_\allowbreak articles\_\allowbreak scraped\_at} and \texttt{idx\_\allowbreak articles\_\allowbreak published\_at} (recency ordering, e.g.\ the \texttt{\_recent} fallback in \texttt{api/routes/search.py}), \texttt{idx\_\allowbreak articles\_\allowbreak story} on \texttt{story\_id}, and \texttt{idx\_\allowbreak articles\_\allowbreak region} on \texttt{region} (the governorate map dashboard).

No GIN index backs full-text search: the French full-text fallback (Chapter 2, Figure~\ref{fig:seq-aisearch}) computes its \texttt{tsvector} inline, at query time, rather than against a stored generated column, so a text-search query runs as a sequential scan rather than an index lookup, an acceptable simplicity trade-off at the corpus's current size, and the same generated-column-plus-GIN pattern common in Postgres full-text search remains available later if the corpus grows enough for it to matter.

Two further tables sit outside this central table. \texttt{entities} (\texttt{id}, \texttt{article\_id} referencing \texttt{articles(id)} \texttt{ON DELETE CASCADE}, \texttt{text}, \texttt{label}) is provisioned for named-entity output, one row per extracted entity per article, with the cascade delete keeping it consistent if an article row is ever removed, but as of this schema no pipeline step inserts into it; it is forward-provisioned schema for a feature not yet wired into \texttt{transform\_articles}, kept here because Figure~\ref{fig:db-schema} already documents it as part of the storage layer. \texttt{pipeline\_metrics} (\texttt{id}, \texttt{run\_at}, \texttt{source}, \texttt{articles\_new}, \texttt{n\_topics}, \texttt{outlier\_pct}, \texttt{sentiment\_pos}\allowbreak/\allowbreak\texttt{neu}\allowbreak/\allowbreak\texttt{neg}, \texttt{embed\_model}), by contrast, is actively written: \texttt{etl/metrics.py}'s \texttt{record\_pipeline\_metrics} inserts one row at the end of every \texttt{run\_pipeline} call (Chapter 2, Figure~\ref{fig:seq-pipeline}). It exists to answer ``what happened in run N'' for monitoring rather than to serve article reads, so its use is covered where it is exercised operationally, in Chapter 7.

\section{Vector storage with pgvector}
\label{sec:pgvector}
The \texttt{embedding} column's dimension, 384, is fixed by the embedding model, \texttt{intfloat/\allowbreak multilingual-e5-small} (\texttt{EMBED\_MODEL\_NAME} in \texttt{etl/transform.py}), a multilingual sentence-transformers model chosen so the same encoder handles French and Arabic source text without a separate model per language. \texttt{model.encode(..., normalize\_embeddings=True)} is called with \texttt{normalize\_embeddings=True}, so every stored vector is unit-length, which is what makes pgvector's cosine-distance operator directly meaningful on the raw column.

The e5 model family distinguishes what is being embedded through a text prefix: \texttt{transform\_articles} (\texttt{etl/transform.py}) prepends \texttt{"passage: "} to every article's title and content and batch-encodes the whole run in one \texttt{model.encode(...)} call at ingestion time. The same prefix is also exposed as a single-text helper, \texttt{get\_passage\_embedding}, used by the project's tests. \texttt{get\_embedding} instead prefixes with \texttt{"query: "} and is what \texttt{api/routes/search.py} calls on an incoming search string. This asymmetric prefixing matches how the model was trained and is what makes a stored passage vector and a live query vector comparable in the same embedding space.

At query time, \texttt{api/routes/search.py}'s \texttt{semantic\_search} first checks \texttt{embeddings\_available()}, a cheap, offline check for whether the model weights are already in the local Hugging Face cache, avoiding a multi-minute download blocking a search request (already described in Chapter 2). When the model is available, the query is embedded with the \texttt{"query: "} prefix, formatted as a pgvector literal, and passed into the query shown in Listing~4.2: \texttt{<=>} is pgvector's cosine-distance operator (0 for identical direction, larger for more dissimilar vectors), used both to order the results and, subtracted from 1, to report a similarity score back to the frontend. The same \texttt{embedding <=> ...} pattern, gated behind the same \texttt{embeddings\_available()} check, is reused by \texttt{api/routes/genai.py} to retrieve the five closest articles as grounding context for the ``ask the news'' chat assistant (Chapter 2): one column and one operator serving two features.

\begin{verbatim}
SELECT id, url, source, title, sentiment, topic_label,
       1 - (embedding <=> %s::vector) AS similarity
FROM articles
WHERE embedding IS NOT NULL
ORDER BY embedding <=> %s::vector
LIMIT %s
\end{verbatim}
\begin{center}
\textit{Listing~4.2: pgvector cosine-distance search (\texttt{semantic\_search}, \texttt{api/routes/search.py}): \texttt{<=>} ranks articles by embedding similarity to the query vector.}
\end{center}

The \texttt{WHERE embedding IS NOT NULL} guard filters out rows the pipeline has not yet enriched: raw articles saved by the lightweight, ML-free load path (\texttt{save\_raw\_articles}, Chapter 3) have no embedding until the full pipeline's transform step runs over them. \texttt{db/\allowbreak schema.sql} declares no ANN index (no \texttt{ivfflat} or \texttt{hnsw} index) on \texttt{embedding}, so every similarity query above runs pgvector's exact, brute-force distance scan over all non-null embeddings rather than an approximate lookup. Combined with the scale argument from Section~\ref{sec:db-tech}, this is a deliberate choice rather than an oversight: an ANN index trades a small amount of recall for query speed at large scale, a trade not worth making yet at a few thousand rows, and one that can be added later with a single \texttt{CREATE INDEX ... USING ivfflat} statement without changing any application code, since it only changes how Postgres executes the same \texttt{<=>} query, not the query itself.

\section{Migrations}
\label{sec:migrations}
The schema did not arrive in its current shape in one step. \texttt{db/migrations/001\_news\_hub.sql} is an additive migration meant to be run against an already-deployed database: every statement is either \texttt{ADD COLUMN IF NOT EXISTS} or \texttt{CREATE ... IF NOT EXISTS}, so re-running it against a database that already has these changes is a no-op rather than an error. It adds the \texttt{published\_at}, \texttt{image\_url}, \texttt{categories}, \texttt{summary}, and \texttt{story\_id} columns to \texttt{articles} (the column set behind the article-preview and summary features described in Chapter 2) together with their two indexes, and it creates the \texttt{entities} and \texttt{pipeline\_metrics} tables (Section~\ref{sec:schema}).

\texttt{db/migrations/002\_region.sql} is narrower, adding exactly the \texttt{region} column and its index for the Tunisia governorate map feature:

\begin{verbatim}
ALTER TABLE articles ADD COLUMN IF NOT EXISTS region TEXT;
CREATE INDEX IF NOT EXISTS idx_articles_region ON articles(region);
\end{verbatim}
\begin{center}
\textit{Listing~4.3: region-tagging migration (\texttt{db/migrations/002\_region.sql}): a nullable column, \texttt{NULL} meaning an article the region tagger could not confidently place in one of Tunisia's governorates.}
\end{center}

\texttt{db/schema.sql} itself already contains the union of the base schema and both migrations: every column above appears directly in its \texttt{CREATE TABLE IF NOT EXISTS articles} block, so it is not a hand-maintained snapshot that could drift from the migration history but the same additive statements folded into one script. A fresh environment bootstraps straight from \texttt{schema.sql}; an existing deployment instead applies the two migrations, in order, to converge on the same final shape without ever dropping data (Chapter 9 covers how this runs inside the Docker Compose stack). Every \texttt{ALTER} across both migrations is \texttt{ADD COLUMN}, never a drop or a rename, so the two layer cleanly on the original schema without needing a migration framework such as Alembic or Flyway, a deliberate simplicity choice for a project at this size and history, at the cost of no automatic down-migration and no separate schema-version tracking table.

\section{Conclusion}
This chapter opened the database behind Chapter 2's class diagram: a single PostgreSQL instance, extended with pgvector, storing both an article's structured metadata and its 384-dimension multilingual embedding in one \texttt{articles} table, indexed for the access patterns the dashboard and API actually exercise, alongside a forward-provisioned \texttt{entities} table and an actively written \texttt{pipeline\_metrics} table. pgvector's \texttt{<=>} cosine-distance operator is the one primitive both semantic search and the chat assistant's retrieval step build on, run as an exact scan deliberately rather than through an approximate index, at a corpus size where that trade is not yet needed. Two small, additive migrations grew the schema to its current shape without a migration framework and without ever touching data already collected. The next chapter follows the \texttt{embedding}, \texttt{sentiment}, \texttt{topic\_id}, and \texttt{region} columns introduced here into the machine learning pipeline that actually computes them.
