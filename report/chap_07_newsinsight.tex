\chapter{Orchestration and Experiment Tracking}

\section{Introduction}
\label{sec:orch-intro}
\begin{sloppypar}
Chapter 1 named \emph{reproducible MLOps} as one of News Insight's four founding objectives (wrapping the pipeline in tooling that makes it observable and repeatable rather than a collection of ad hoc scripts) and listed Prefect and MLflow among the project's core technology choices (Table~\ref{tab:tech-stack}). Chapter 2's architecture overview already named both services by their exact role (Prefect wrapping the scrape-transform-load sequence in one flow, \texttt{news-pipeline}, and MLflow logging every BERTopic run under the \texttt{news-clustering} experiment), and its sequence diagram (Figure~\ref{fig:seq-pipeline}) already traced one full flow run down to the task-level retry. Every chapter since has deferred a piece of the operational story to this one: Chapter 3's error-handling section named Prefect's task retry as ``the outermost safety net'' (Section~\ref{sec:error-handling}) without describing how it is configured; Chapter 5 named all three ways \texttt{run\_pipeline} is reachable, including ``wrapped in a Prefect task with two retries and a 60-second delay (\texttt{flows/\allowbreak news\_flow.py}, Chapter 7)'', and closed both its BERTopic section and its metrics section by pointing the reader here (Sections~\ref{sec:bertopic} and~\ref{sec:metrics}); and Chapter 6 ended by handing off exactly ``how the whole pipeline that feeds them is orchestrated and observed in production: Prefect flows and MLflow experiment tracking.'' This chapter reads that operational layer end to end: the flow and task definitions in \texttt{flows/\allowbreak news\_flow.py}, the cron deployment in \texttt{flows/\allowbreak schedule.py}, the MLflow logging call in \texttt{etl/\allowbreak cluster.py}, and the two Docker Compose services and environment variables, \texttt{PREFECT\_API\_URL} and \texttt{MLFLOW\_TRACKING\_URI}, that connect the two.
\end{sloppypar}

\section{Orchestration technology choice}
\label{sec:orch-choice}
\begin{sloppypar}
\texttt{requirements.txt} pins \texttt{prefect>=3.0.0}; Chapter 1's technology table already listed the choice at a glance (``Prefect: scheduling and running the scrape / transform / cluster pipeline'') without arguing for it over the field's other well-known option, Apache Airflow. The two occupy different points on the same trade-off Chapter 3 made when it picked \texttt{requests}+\texttt{BeautifulSoup} over a full browser-automation stack: simplicity against the ability to express a bigger problem. Airflow is built around a DAG-of-operators model, aimed at coordinating many discrete, interdependent tasks across a schedule, and it pays for that with its own always-on scheduler process, a metadata database, and tasks authored against its own operator API rather than as plain Python functions. News Insight's pipeline, by contrast, is the single straight-line sequence Chapter 2 already traced (extract, transform, load, strictly in that order, per Section~\ref{sec:pipeline-overview}), with no branching and no cross-task fan-out to schedule around. Prefect's \texttt{@flow}/\texttt{@task} decorators (Section~\ref{sec:flow-design}) let that existing sequence, \texttt{etl.pipeline.run\_pipeline}, become a scheduled, retried, and observable unit without being rewritten against a different execution model at all: two decorators on top of code that would otherwise just be called directly, which is the right amount of orchestration machinery for one linear job run by a small team, rather than the DAG-authoring and infrastructure overhead a many-task, many-dependency system like Airflow is built to justify.
\end{sloppypar}

\section{Flow design}
\label{sec:flow-design}
\begin{sloppypar}
Figure~\ref{fig:prefect-flow} is the map this section and the next two follow: the two ways the pipeline starts (a cron-scheduled deployment, Section~\ref{sec:scheduling}, or a direct HTTP trigger, below), the flow/task pair inside \texttt{flows/\allowbreak news\_flow.py}, the three-step \texttt{run\_pipeline} call it wraps (already read in full in Chapter 5, Section~\ref{sec:pipeline-overview}), and the two places a run becomes observable afterward: the Prefect UI and the \texttt{pipeline\_metrics} table (Section~\ref{sec:monitoring}).
\end{sloppypar}

\begin{figure}[H]
\centering
\includegraphics[width=0.85\columnwidth]{img/prefect-flow.png}
\caption{Orchestration map: how a run starts (cron deployment or manual HTTP trigger), the \texttt{news-pipeline} flow and its single retried task, the \texttt{run\_pipeline} steps it wraps, and the two observability endpoints (the Prefect UI and the \texttt{pipeline\_metrics} table) covered later in this chapter.}
\label{fig:prefect-flow}
\end{figure}

\begin{sloppypar}
\texttt{flows/\allowbreak news\_flow.py} (Listing~7.1) defines exactly one task and one flow. \texttt{pipeline\_task}, decorated \texttt{@task(name="run-news-pipeline", retries=2, retry\_delay\_seconds=60)}, does nothing but call \texttt{run\_pipeline(max\_pages=max\_pages)} and return its result dictionary. The whole three-step extract/transform/load sequence is wrapped by a single Prefect task, not decomposed into one task per step. \texttt{news\_flow}, decorated \texttt{@flow(name="news-pipeline", description="Scrape, transform and load Tunisian news articles")}, logs a start message, calls \texttt{pipeline\_task(max\_pages=max\_pages)}, logs the returned result, and returns it; its own \texttt{max\_pages} parameter defaults to \texttt{5}. Because the task boundary sits around the entire \texttt{run\_pipeline} call rather than around each of its three steps, Prefect's retry is coarse-grained by design: if \texttt{transform\_articles} raises after \texttt{run\_extraction} already succeeded, the retry re-runs the whole task, extraction included, rather than resuming from the failed step. That repeated work is harmless rather than destructive because of a guarantee from two chapters back (\texttt{save\_articles}'s \texttt{INSERT ... ON CONFLICT (url) DO UPDATE}, Chapter 2), so re-scraping already-seen URLs on a retried attempt refreshes their rows instead of duplicating them.
\end{sloppypar}

\begin{verbatim}
@task(name="run-news-pipeline", retries=2, retry_delay_seconds=60)
def pipeline_task(max_pages: int) -> dict:
    return run_pipeline(max_pages=max_pages)

@flow(
    name="news-pipeline",
    description="Scrape, transform and load Tunisian news articles",
)
def news_flow(max_pages: int = 5):
    logger.info("Starting news pipeline flow")
    result = pipeline_task(max_pages=max_pages)
    logger.info(f"Flow completed: {result}")
    return result
\end{verbatim}
\begin{center}
\textit{Listing~7.1: the flow and its one task (\texttt{flows/news\_flow.py}): \texttt{run\_pipeline}'s three-step sequence (Chapter 5) is wrapped by a single retried task, not split one task per step.}
\end{center}

\begin{sloppypar}
Two entry points can start this flow outside its cron schedule. \texttt{POST /pipeline/trigger} (\texttt{trigger\_pipeline}, \texttt{api/\allowbreak routes/\allowbreak pipeline.py}) hands \texttt{news\_flow} to FastAPI's \texttt{BackgroundTasks.add\_task}, so the flow runs in-process after the HTTP response has already been sent back to the caller; its own \texttt{max\_pages} default is \texttt{3}, smaller than the flow function's own default of \texttt{5} and smaller than the daily deployment's parameter (Section~\ref{sec:scheduling}), a lighter default for what is more often a quick manual check than the full daily batch. Because \texttt{news\_flow} is itself a Prefect \texttt{@flow}, calling it this way (directly, from a background task, with no deployment or worker involved) still emits a flow run to whichever server \texttt{PREFECT\_API\_URL} points at, so both the cron-scheduled path and this on-demand path end up as visible runs in the same Prefect UI (Figure~\ref{fig:screen-prefect}). \texttt{POST /pipeline/cluster} (\texttt{trigger\_clustering}) is the pipeline's other background-task endpoint, but it calls \texttt{run\_clustering} (\texttt{etl/\allowbreak cluster.py}) directly, a plain function, carrying no \texttt{@task} or \texttt{@flow} decorator at all. Clustering runs, whether started this way or from \texttt{scripts/\allowbreak cluster.py} (\texttt{make cluster}), therefore never appear in the Prefect UI; MLflow's \texttt{news-clustering} experiment (Section~\ref{sec:mlflow-tracking}) is their only observability surface. The two halves of the pipeline are consequently watched by two different tools, each blind to the other's runs.
\end{sloppypar}

\begin{sloppypar}
One naming overlap is worth flagging so it does not read as a contradiction later in this chapter: the Makefile's own \texttt{make pipeline} target (\texttt{scrape} then \texttt{transform} then \texttt{cluster}, three standalone scripts run locally, per the README) is a different sequence from the Prefect flow this section describes. The Prefect flow is named \texttt{news-pipeline}, but it only ever wraps \texttt{run\_pipeline} (extract, transform, load) and never runs BERTopic clustering; the two ``pipelines'' share a name in casual usage but not a code path.
\end{sloppypar}

\section{Scheduling}
\label{sec:scheduling}
\begin{sloppypar}
\texttt{flows/\allowbreak schedule.py} registers exactly one deployment via \texttt{Deployment.\allowbreak build\_from\_flow}: flow \texttt{news\_flow}, name \texttt{"daily-news-pipeline"}, schedule \texttt{CronSchedule(cron="0 6 * * *", timezone="Africa/Tunis")}, and \texttt{parameters=\{"max\_pages": 5\}}, followed by \texttt{deployment.apply()} when the module is run as a script. The cron expression fires once a day at 06:00, in the \texttt{Africa/Tunis} timezone specifically rather than whatever timezone the server's own clock happens to be in (ahead of the Tunisian news day, so a run picks up whatever was published overnight before readers open the dashboard in the morning), and always with the same \texttt{max\_pages=5} the flow's own default already uses. Running \texttt{python -m flows.schedule} is a registration step, not a long-running process: it pushes this deployment's definition to whichever Prefect server \texttt{PREFECT\_API\_URL} names, and only needs re-running when the schedule or its parameters change.
\end{sloppypar}

\begin{sloppypar}
Registering a deployment and actually executing it on schedule are two different things in Prefect 3.x, and only the first of the two is wired up in this project as it stands. \texttt{docker-compose.yml}'s \texttt{prefect} service runs \texttt{prefect server start -{}-host 0.0.0.0} (the API and UI process a deployment is registered \emph{against}), but executing a deployment's scheduled runs needs a separate worker process polling a work pool for it, and no such worker or agent service is defined anywhere in \texttt{docker-compose.yml}. Neither the Makefile, \texttt{docker-compose.yml}, nor the README's run instructions ever invoke \texttt{flows/\allowbreak schedule.py} either. A \texttt{daily-news-pipeline} deployment, once registered, would appear in the Prefect UI's deployment list with its cron schedule attached; nothing in the current containerized stack polls and executes it automatically. In today's setup, the two paths that actually run \texttt{news\_flow} are the two direct ones already covered: a manual \texttt{python -m flows.\allowbreak news\_flow} invocation (the module's own \texttt{\_\_main\_\_} block, \texttt{max\_pages=1}), and the on-demand \texttt{POST /pipeline/trigger} endpoint, both calling the flow directly, bypassing the scheduled deployment entirely.
\end{sloppypar}

\section{Experiment tracking with MLflow}
\label{sec:mlflow-tracking}
\begin{sloppypar}
\texttt{log\_clustering\_run(n\_topics, outlier\_pct, min\_cluster\_size)} (\texttt{etl/\allowbreak cluster.py}) is \texttt{run\_clustering}'s last step (Chapter 5, Section~\ref{sec:bertopic}), called once every BERTopic run completes and \texttt{update\_topics} has already written \texttt{topic\_id}/\texttt{topic\_label} back to the \texttt{articles} table. It lazily imports \texttt{mlflow} into a module-level placeholder on first call (the same lazy-import discipline Chapter 5 and Chapter 6 already described for the embedding, sentiment, and LLM clients, keeping \texttt{import etl.cluster} light for any process that never actually clusters), then points the client at \texttt{os.getenv("MLFLOW\_TRACKING\_URI", "http://localhost:5000")} and calls \texttt{mlflow.set\_experiment("news-clustering")}, so every clustering run, regardless of which of the entry points in Section~\ref{sec:flow-design} started it, lands in the same named experiment. Inside \texttt{mlflow.start\_run()} (Listing~7.2), \texttt{log\_params} records \texttt{embed\_model} (hard-coded here as the literal string \texttt{"intfloat/multilingual-e5-small"} rather than imported from the \texttt{EMBED\_MODEL} constant Chapter 5 named in \texttt{etl/\allowbreak pipeline.py}, Section~\ref{sec:pipeline-overview}, the same value duplicated rather than shared) and \texttt{min\_cluster\_size}, the caller's own parameter (\texttt{3} by default, Section~\ref{sec:bertopic}). \texttt{log\_metrics} records \texttt{n\_topics} and \texttt{outlier\_pct}, computed exactly as Chapter 5 described: the count of non-outlier BERTopic clusters, and the fraction of articles assigned topic \texttt{-1}.
\end{sloppypar}

\begin{sloppypar}
The saved model directory is logged last, and not with a single unconditional call: \texttt{(mlflow.log\_artifacts if os.path.isdir("models/\allowbreak bertopic\_model") else mlflow.log\_artifact)("models/\allowbreak bertopic\_model")} picks between MLflow's directory-logging and single-file-logging functions at call time. This is itself the fix for a real bug, recorded in commit \textit{fix: timeline days lower bound + robust mlflow artifact logging}: the original code called \texttt{log\_artifacts} (plural) unconditionally, but BERTopic's own \texttt{.save()} may write a single file rather than a directory depending on its serialization backend, and \texttt{log\_artifacts} cannot accept a bare file path, so a save that happened to produce one file would fail this last logging step outright. \texttt{run\_clustering} wraps the entire \texttt{log\_clustering\_run} call in its own \texttt{try/except}, logging a warning on failure rather than raising, so (exactly like the asymmetric degrade-instead-of-fail pattern Chapter 6 described for LLM calls) an MLflow outage costs only the tracking record for that run, never the clustering result already committed to \texttt{articles.topic\_id}/\texttt{topic\_label}.
\end{sloppypar}

\begin{verbatim}
mlflow.set_tracking_uri(os.getenv("MLFLOW_TRACKING_URI", "http://localhost:5000"))
mlflow.set_experiment("news-clustering")
with mlflow.start_run():
    mlflow.log_params({"embed_model": "intfloat/multilingual-e5-small",
                       "min_cluster_size": min_cluster_size})
    mlflow.log_metrics({"n_topics": n_topics, "outlier_pct": outlier_pct})
    (mlflow.log_artifacts if os.path.isdir("models/bertopic_model")
     else mlflow.log_artifact)("models/bertopic_model")
\end{verbatim}
\begin{center}
\textit{Listing~7.2: logging one clustering run (\texttt{log\_clustering\_run}, \texttt{etl/cluster.py}): parameters, metrics, and the fitted model as an artifact, all under the \texttt{news-clustering} experiment.}
\end{center}

\begin{sloppypar}
\texttt{MLFLOW\_TRACKING\_URI}'s code-level default, \texttt{http://localhost:5000}, matches the README and \texttt{.env.example}'s own documented default for local, non-Docker use. Inside \texttt{docker-compose.yml}, the \texttt{api} container overrides it to \texttt{http://mlflow:5000} (the \texttt{mlflow} service's container-internal port, reached by its Compose service name over the internal network), while the same server is reachable from a browser on the host at \texttt{http://localhost:5001}, since the \texttt{mlflow} service's own port mapping, \texttt{"5001:5000"}, exists specifically to avoid a host port already claimed by another local service (the file's own comment names macOS AirPlay as the usual culprit). Figure~\ref{fig:screen-mlflow} is one such run, viewed through that host-mapped UI.
\end{sloppypar}

\begin{sloppypar}
Chapter 5 already noted, from the \texttt{pipeline\_metrics} side, that \texttt{n\_topics} and \texttt{outlier\_pct} stay \texttt{None} on every row \texttt{run\_pipeline} writes, because clustering is a separate stage that never runs inside \texttt{run\_pipeline} itself (Section~\ref{sec:metrics}). This section closes that loop from the MLflow side: those two values are not simply missing from the system, they are tracked in a different place entirely, one row per BERTopic run under the \texttt{news-clustering} experiment rather than one row per \texttt{pipeline\_metrics} insert. ``Was the last pipeline run healthy'' and ``was the last clustering run healthy'' are, as a result, two separate questions answered by two separate stores (PostgreSQL for the first, MLflow for the second) that never populate the same row.
\end{sloppypar}

\section{Monitoring and reliability}
\label{sec:monitoring}
\begin{sloppypar}
\texttt{docker-compose.yml}'s health configuration mirrors the failure-handling asymmetry the code itself already encodes. The \texttt{api} service's \texttt{depends\_on} only requires \texttt{postgres} and \texttt{prefect} to report \texttt{service\_healthy} (each backed by its own \texttt{healthcheck} block, \texttt{pg\_isready} for Postgres and a small \texttt{urllib}-based request against \texttt{/api/health} for Prefect), while \texttt{mlflow} carries no \texttt{healthcheck} at all and is absent from \texttt{api}'s \texttt{depends\_on} list entirely. A slow-starting or crashed MLflow container therefore never blocks the API from starting, which matches exactly what the code already guarantees at the call-site level (Section~\ref{sec:mlflow-tracking}): MLflow is wired in as a best-effort side channel, not a dependency the rest of the system waits on or fails without.
\end{sloppypar}

\begin{sloppypar}
That same fail-loud/fail-quiet split runs through \texttt{run\_pipeline} itself (\texttt{etl/\allowbreak pipeline.py}, Chapter 5). Its three core steps (\texttt{run\_extraction}, \texttt{transform\_articles}, \texttt{save\_articles}) carry no \texttt{try/except} of their own at that level; an exception in any of them propagates all the way up through \texttt{pipeline\_task} to Prefect, which is exactly what makes the task-level retry (Section~\ref{sec:flow-design}) the mechanism that handles it, two attempts, 60 seconds apart, rather than a swallowed error. Only the very last step, \texttt{record\_pipeline\_metrics}, is wrapped in its own \texttt{try/except} (the pipeline's own comment reads ``never break the pipeline on a write failure''), so a database hiccup while writing the \texttt{pipeline\_metrics} row costs only that row, not the scrape/transform/load work already committed ahead of it. \texttt{run\_clustering} repeats the same pattern twice over: the LLM topic-labelling call falls back to raw BERTopic names on failure (Chapter 5), and \texttt{log\_clustering\_run} falls back to a logged warning (Section~\ref{sec:mlflow-tracking}): every side channel degrades quietly, while the two things each run actually exists to produce, saved articles and labelled topics, either succeed outright or fail loud enough for Prefect's retry, or the operator watching the logs, to notice.
\end{sloppypar}

\begin{sloppypar}
Put together, News Insight's operational visibility rests on three surfaces, each already introduced from a different angle in an earlier chapter: the Prefect UI (Figure~\ref{fig:screen-prefect}) for flow- and task-run history and retries; the MLflow UI (Figure~\ref{fig:screen-mlflow}) for clustering-run parameters and metrics; and the \texttt{pipeline\_metrics} table (Chapter 4, Section~\ref{sec:schema}) as the one destination queryable directly with SQL, outside either tool's own UI or retention window, for the scrape/transform/load side of the pipeline.
\end{sloppypar}

\section{Conclusion}
\begin{sloppypar}
This chapter closed the operational story every earlier chapter had deferred to it. A single Prefect flow, \texttt{news-pipeline}, wraps \texttt{run\_pipeline} in one retried task, \texttt{run-news-pipeline} (two attempts, 60 seconds apart); a registered daily deployment names the schedule (06:00, \texttt{Africa/Tunis}) that the flow is meant to run on, though, as read directly from \texttt{docker-compose.yml}, no worker process in the current stack is actually configured to execute it, leaving the manual CLI invocation and the \texttt{POST /pipeline/trigger} endpoint as the paths that run today. Every BERTopic run is logged separately, under MLflow's \texttt{news-clustering} experiment, with the same parameters and metrics Chapter 5 introduced and this chapter finally showed being written and displayed. Across both services and the pipeline code they wrap, the same reliability posture repeats: the steps that actually produce data (scraping, transforming, saving, clustering) fail loud, letting Prefect's retry or an operator's attention handle it, while every side channel that only records \emph{what happened} (a \texttt{pipeline\_metrics} row, an MLflow run) fails quiet, so a monitoring outage never becomes a data outage. The next chapter turns from this orchestration layer to the FastAPI backend it and every other feature in this report sit behind (the routes already named in passing across five chapters, from \texttt{/pipeline} to \texttt{/genai} to \texttt{/search}) as one coherent surface.
\end{sloppypar}
